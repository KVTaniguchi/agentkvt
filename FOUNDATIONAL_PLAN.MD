1. System Architecture Overview
Build a localized, closed-loop agentic system consisting of two primary components:

The Brain (macOS): A headless background service designed for Apple Silicon (e.g., Mac Studio). It acts as an MCP (Model Context Protocol) Host and autonomous task runner. Inference is handled locally via an LLM host (like Ollama or LM Studio).

The Remote (iOS): A SwiftUI dashboard running on an iPhone. It contains zero chat interfaces. It strictly displays dynamic AppIntentButtons generated by the Mac agent's background research.

The Bridge: Shared state managed entirely through SwiftData (synchronized via local network/Tailscale or CloudKit). The macOS app acts as the writer/processor, and the iOS app acts as the observer/controller.

2. Hardware Resource Constraints (Critical Setup)
The macOS agent must be configured to prevent system lockups during deep reasoning loops, ensuring background OS tasks (like real-time audio dictation via Wispr Flow) do not crash.

LLM Throttling: The AI editor must generate configuration scripts or documentation for the local LLM host to cap GPU and Neural Engine usage at 80%.

Preserved Overhead: A strict 20% compute overhead must remain unallocated at all times to ensure system stability during heavy agentic while loops.

3. Database Schema (SwiftData)
Create a shared Swift Package (ManagerCore) containing the following highly structured entities.

LifeContext: Static facts and user preferences the agent must consult before taking action (e.g., specific goals, geographic locations, important dates).

MissionDefinition: The configuration models for user-defined background tasks (detailed in Section 5).

ActionItem: The dynamic buttons for the iOS app. Properties include title, systemIntent, payloadData, relevanceScore, and timestamp.

AgentLog: A transparent, append-only ledger of what the agent "thought," what tools it executed, and the outcome, allowing the user to audit the agent's behavior.

4. Agent Sandboxing & MCP Tool Guardrails
To prevent destructive actions during autonomous loops, implement strict boundaries in the MCP Server layer. All tools must be built with "Zero Trust" principles regarding the LLM's output.

The "Only Me" Notification Tool: Create a Swift tool called sendNotificationEmail. Hardcode the destination email address (the user's address) directly into the compiled function or secure keychain. Expose only subject and body parameters to the LLM. It must be physically impossible for the agent to email external addresses.

Restricted Version Control Tool: Create a GitHub MCP tool. It must authenticate using a dedicated Bot Personal Access Token (PAT). This token configuration must enforce scoping exclusively to designated agent-accessible repositories, completely isolating the user's primary GitHub profile and private codebases.

5. Core Autonomous Missions (Dynamic Mission Engine)
Do not hardcode specific user tasks. Instead, build a Dynamic Mission Engine that allows the user to define, schedule, and execute custom background loops. The macOS service will act as a runner that processes these user-defined missions.

A. The MissionDefinition Schema
Users construct their own agents from the iOS interface. The entity must include:

missionName: String identifier (e.g., "Tech Job Scout", "Budget Sentinel").

systemPrompt: The user-defined instructions for the LLM (e.g., "Search for iOS roles within 15 miles and calculate the commute.").

triggerSchedule: How often the Mac agent runs this loop (e.g., Daily at 8 AM, Weekly, or Triggered by a webhook/external event).

allowedMCPTools: An array of tool IDs this specific mission is authorized to use (e.g., [web_scraper, maps_api, pdf_generator]), ensuring the principle of least privilege.

B. The Mac Execution Loop
The background macOS app must run a CRON-style scheduler that observes the MissionDefinition table.

Logic: When a mission is triggered, the Mac app passes the systemPrompt and the allowedMCPTools to the local LLM.

Action: The LLM executes the requested reasoning and tool calls within its sandbox.

Output: Regardless of the custom mission, the output must always be standardized. The agent must write the result to the shared ActionItem table, generating a dynamic SwiftUI button for the iOS dashboard (e.g., "Review New Job Leads," "Approve Generated Document").

C. Example Implementation (For Testing Purposes Only)
To verify the engine works, the developer will input the following test missions via the UI, but they must not be hardcoded into the compiled binary:

Test Mission 1 (Career): A prompt instructing the agent to scan job boards and cross-reference a local resume file, generating an ActionItem for high-match roles.

Test Mission 2 (Finance): A prompt instructing the agent to monitor specific CSV transaction exports for impulse purchases, generating an ActionItem for manual review if limits are exceeded.
